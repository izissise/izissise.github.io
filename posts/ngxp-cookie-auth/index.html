Nginx Explorer - Cookie Authentication
Posted on 3024-11-03 DRAFT 2 minutes read

Previously we listed directories with nginx on an html page, now we would like restrict access to directories and files, let’s implement cookie-based authentication.

Basic auth

The simplest form of authentification on the web is basic authentication. It require that all authenticated requests has an Authorization header, but html and browsers do not provides a way to set an Authorization for links in <a> tags. Browsers instead send previously set cookies with every requests on the same site.

We're going to add a new login endpoint using nginx basic auth to check user/password pair and set a cookie for authorized users.

...
server {
    ...
    location @login_success {
        add_header Set-Cookie "ngxp=???; max-age=15552000; path=/; SameSite=Lax; Secure; HttpOnly";
        return 200 "login success";
    }
    location ^~ /___ngxp/login {
        auth_basic "Login";
        auth_delay 1s;
        auth_basic_user_file /opt/ngxp/basic.htpasswd;
        try_files _ @login_success;
    }
}

We can use openssl CLI to create user/password entries into basic.htpasswd file:

printf '%s:%s\n' "$user" "$(openssl passwd -5 "$password")" >> basic.htpasswd

Cookie

We can successfully login using basic auth on the login endpoint!

To check in the listing endpoint that the user is authorized to access the path, we need to store the following in the cookie:

  • username
  • a secret, so the cookie cannot be hand crafted to get access
  • the path base the user has access to

We can use nginx map module to create the information triplet for our users:

#     input     output
map $username $user_cookie {
    default "";
    admin   admin:d31d6b17bbc4df3ccb78d99db085e9d347b3e042e65d5878:/;
    alice   alice:2406b10bfd4c356394a647d5657bde937124f602198f20f3:/alicefiles;
}

We can fill up @login_success ngxp cookie value, by first setting $username variable, $user_cookie will have the corresponding value from the map.

    location @login_success {
        set $username $remote_user; # $remote_user is the basic_auth user in Authorization http header
        add_header Set-Cookie "ngxp=$user_cookie; max-age=15552000; path=/; SameSite=Lax; Secure; HttpOnly";
        return 200 "login success";
    }

Is this cookie authorized?

The only thing left to do is to check our cookie is valid and can access the current uri when listing files. $cookie_ngxp contains the client provided cookie, it is an unstrusted user input, so we should check that it is exactly the same string has the one set for the username.

# get username from http request's cookie
map $cookie_ngxp $username {
    ~^(?<var>[^:]+):.* $var;
    default            "";
}
map ${cookie_ngxp}?${user_cookie} $user_cookie_equal {
    ~^([^\?]+)\?\1$   1; # if $cookie_ngxp == $user_cookie
    default           0;
}

Modifying location / a bit, we can already restrict listing and download for requests with a valid cookie:

...
server {
    ...
    location / {
        if ($user_cookie_equal = 0) { return 401; }
        ...
    }
}

We also want to have access right per path, we first extract the path from $user_cookie:

...

map $user_cookie $user_cookie_access {
    ~^([^:]+):([^:]+):(?<var>[^:]+)$ $var;
    default                          "";
}

And using a map with two variable as input, we can match that $uri starts with the extracted path

...
map ${user_cookie_access}?${uri} $uri_access {
    ~^([^\?]+)\?\1.*$ 1; # if $uri startswith $user_cookie_access
    default           0;
}

Finally we can check that the received cookie is equal to the sent one and that the current path starts with the path stored in the cookie:

...
map ${user_cookie_equal}${uri_access} $user_authorized {
    ~^11$       1; # everything good authorize
    default     0;
}

Going back to the file listing endpoint, we check $user_authorized:

...
server {
    ...
    location / {
        if ($user_authorized = 0) { return 401; }
        ...
    }
}

We now have per user and path accesses!
